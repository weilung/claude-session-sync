# 安全機制白話：它憑什麼保證不弄丟我的資料？

> 這是整份指南**最重要**的一篇。你最擔心的一定是：
> 「AI 幫我寫的這套東西，會不會哪天把我重要的對話或 memory 蓋掉、弄丟？」
>
> 這篇用「**怕什麼 → 怎麼防 → 舉例**」的方式，把每一道防線講白。
> 全部合起來就是一句設計鐵則：**寧可多留、多問、多報告，也絕不擅自覆蓋或刪除。**

---

## 兩條最高鐵則（先記這兩條）

工具內部把所有安全設計濃縮成兩條「無論如何都不能違反」的原則：

- **鐵則 A3 —「不復活、也不擅自刪除」**
  你在某台電腦**刪掉**的東西，不會因為別台還留著就**偷偷跑回來**（復活）；
  反過來，工具**永遠不會替你刪掉**你在別處的資料——它頂多留一個「刪除標記」通知其他電腦。

- **鐵則 C3 —「絕不覆蓋你本機已有的對話檔」**
  只要你本機已經有某個對話檔，工具**永遠不會直接蓋過去**。
  真的兩邊不一樣時，它會**另存一份**（keep-both），把選擇權留給你。

底下每一道防線，都是在落實這兩條。

---

## 一、tombstone（刪除標記）——防「刪掉的東西又跑回來」

**怕什麼？**
假設你在家裡電腦刪掉一則過期（甚至含**敏感資訊**）的對話或 memory。
下次同步時，公司電腦上還留著那一份——如果工具傻傻地「哪邊有就補到哪邊」，
那則你**刻意刪掉**的東西就會**又被同步回來**，你得一台一台刪，永遠刪不乾淨。

**怎麼防？**
當你刪掉一份東西、工具偵測到「這是**曾經存在、現在被刪掉**的」，
它會在 hub 放一個小小的**刪除標記（tombstone）**，意思是「這份是被人**故意**刪的，別再補回來」。
其他電腦看到這個標記，就知道**不要復活**它。

- 刪除標記**只記錄「被刪了」這件事**，不會去動你別台電腦上還留著的那一份（那是鐵則 A3：工具不替你刪）。
- 如果一邊刪了、另一邊卻**同時改過內容**，工具不會自作主張，而是報成「**刪除 vs 修改衝突**」，交給你決定。

**舉例：**
你在家裡刪掉 `舊密碼筆記.md`。同步後，hub 出現一個標記說「這份被刪了」。
公司電腦下次同步時，看到標記 → 知道你是**故意**刪的 → **不會**把它補回你家。
（但公司那台上如果還有，工具不會自動幫你刪——你要親自去刪，這是安全保守。）

---

## 二、read-verify-write（寫前重讀、寫後驗證）——防「寫到一半資料錯亂」

**怕什麼？**
工具在「盤算要怎麼同步」到「真的動手寫」之間，有一小段時間差。
萬一這中間檔案又被 Claude 或你改動了，工具卻照著**過期的盤算**去寫，就可能蓋錯。
另外，複製檔案本身也可能出錯（尤其跨作業系統、跨網路碟）。

**怎麼防？**（三重確認）

1. **寫之前**：重新讀一次目標檔現在的內容，跟先前盤算時的快照比對。
   一旦發現「跟我以為的不一樣了」→ **重新判斷**（通常會升級成「這是分岔、要問人」）→ **中止**這個檔，不照舊硬寫。
2. **寫的時候**：先寫到同目錄的一個**暫存檔**，確認寫入落到磁碟，再**一次性改名**成正式檔（rename）。
   這樣即使中途斷電，也不會留下寫一半的壞檔。
3. **寫之後**：把剛寫好的檔**再讀回來**，核對內容雜湊值（等於指紋）跟預期一致，才算數。

**舉例：**
工具盤算「把這則對話從 hub 補到你電腦」。動手前它重讀一次，發現你電腦上這則**剛剛被你自己改過**了 →
它立刻停手，改標成「兩邊都動過、這是分岔」，讓你來決定，**而不是**用 hub 的版本蓋掉你剛打的字。

---

## 三、keep-both（不覆蓋、留兩份）——防「兩邊都改過時互相蓋掉」

**怕什麼？**
最經典的同步災難：你在家改了對話 A，同事在公司也從同一個起點改了對話 A，變成兩個不同的分支（fork）。
笨的同步工具會用其中一版蓋掉另一版，另一版就永遠消失了。

**怎麼防？**
遇到「兩邊從共同起點各自長出不同內容」時，工具**絕不選邊、絕不覆蓋**。
它把 hub 那一份**另存成一個新檔**放到你電腦（這叫 keep-both＝兩份都留），
你電腦原本那份**原封不動**。兩份都在，你再決定怎麼處理。

- 這就是鐵則 C3 的具體做法：**本機已有的對話檔，永遠不被蓋過去。**
- 「往你電腦寫入」一律用「只能新建、不能覆蓋既有」的方式（技術上叫 O_EXCL）：撞名就 keep-both，不覆蓋。

**舉例：**
你家和公司都從同一通對話接著聊了不同內容。同步時工具發現分岔 →
你電腦上留著你家的版本，旁邊多出一個「公司版本」的複製檔 →
你可以自己看兩份、或用下面說的「互動合併」把兩枝併起來。

---

## 四、lock（鎖）——防「多台同時同步互相打架」

**怕什麼？**
自己這群有好幾台電腦共用同一個 hub。如果兩台**同時**在寫同一個 hub 檔案，可能寫出交錯錯亂的內容。

**怎麼防？**
工具在動 hub 上某個專案的資料前，會先搶一把**鎖**（一個只能有一個人拿到的小檔）。
拿到的人才能寫，其他人等一下或稍後再跑。寫完就釋放。

- 鎖只由**放它的人**用一個獨有的識別碼（token）來解除，不會誤解別人的鎖。
- 萬一某台當掉、鎖沒清掉（stale 鎖），可以用 `doctor --break-lock` 安全清除——
  而且它**只清「確定是同一台、而且那個程序已經死掉」的鎖**，跨機器或還活著的鎖絕不亂動。

> **老實說的限制**：網路碟／外接碟（NFS、SMB、exFAT、雲端掛載）的鎖**不是 100% 可靠**。
> 所以最穩的用法是：**同一群裡，同一時間只讓一台電腦跑同步。** 這一點文件裡有明講、不藏。

## 五、bootstrap 基線——防「第一次同步就把一邊誤當成『被刪光』」

**怕什麼？**
工具判斷「東西是被刪的還是根本沒同步過」，靠的是「**我上次記得有哪些**」這份底稿。
如果沒有底稿，第一次同步時，一台有、一台沒有，工具分不清這是「新東西該複製」還是「被刪掉別復活」。

**怎麼防？**
第一次使用要先跑 **bootstrap（建立基線）**：它掃一遍現況，記下「目前雙方各有哪些對話、哪些 memory」當作**起始底稿**。
有了底稿，之後才分得出「新增 / 刪除 / 修改」。

- **沒建過基線的專案，工具採最保守態度**：寧可**擋下來要你先 bootstrap**，也不亂複製、不亂寫刪除標記（避免下次錯誤復活）。
- 這也是為什麼 [指令手冊](03-指令手冊.md) 說「**第一次一定先 bootstrap**」。

**舉例：**
新買一台電腦，`~/.claude` 是空的。直接同步的話，工具可能誤以為「hub 上一堆對話在你這台被刪光了」。
先 bootstrap 建好基線（記下「這台目前就是空的、hub 有這些」）→ 之後同步就會正確地「把 hub 的補過來」，而不是誤判成刪除。

## 六、預設只預覽（dry-run）——你沒說要做，就不做

**怕什麼？**
最單純的安全：不小心跑了指令，就把東西改了。

**怎麼防？**
**幾乎每個會寫檔的指令，預設都只「預覽」**——把「我打算做這些事」印給你看，**但不真的動手**。
你看過、確認沒問題，再加 `--apply`（或 bootstrap 的 `--yes`）才真的落地。

**舉例：**
```
claude-session-sync sync          # 只告訴你「會複製這 3 則、這 1 則兩邊分岔要你決定」——不動任何檔
claude-session-sync sync --apply  # 你確認後，才真的執行
```

## 七、互動合併／挑選——分岔時，把選擇權交給你

**怕什麼？**
兩邊分岔（fork）時，機器**無權**替你決定要哪一版。

**怎麼防？**
偵測到分岔時，工具停下來給你選：

- **union（合併兩枝）**：把兩個分支的對話行**依身分去重後併成一份**，一行都不丟，並標出「目前最新在哪」。
  這是**無損**的併，不是「AI 重寫你的對話」——只是把兩枝安全地接在一起。
  萬一這份對話結構特殊、併起來不安全，工具會**自動退回「挑選」**，不硬併。
- **keep-both（留兩份）**：把 hub 那枝複製到你電腦，兩份都在，你自己看。
- **skip（跳過）**：這次先不處理。

非互動時（沒加 `--interactive`），分岔一律只**報告、保留兩份、不動**，等你有空再來處理。

## 八、memory 特有的兩道保護

memory 是一堆 `.md` 檔，比對話更需要小心，因為它常含**你精煉過、甚至敏感**的資訊。

### 8a. 絕不把兩則「意思不同」的 memory 判成同一則
工具比對 memory 時，用的是**正規化後的內容指紋**：
會吸收掉無意義的差異（換行符、frontmatter 欄位順序、易變欄位如來源 id），
但**絕不壓掉真正的語意差異**。這條是 memory 的最高鐵則——判錯就等於靜默覆蓋，所以寧可謹慎。

> **兩層比對（都守這條鐵則）**：① **自動層＝精確比對**（同 `name`／frontmatter 字面相同才算同一則）——只有這層會影響自動同步。
> ② 「**同一件事、不同檔名（不同 slug）**」另有一個**唯讀輔助工具** `memory-merge --fuzzy`：它只**列出**疑似重複的候選對讓你**人工複核**，
> 分數**永遠不裁定**、**絕不自動合併/覆蓋**；要保留兩版也得你**逐對放行**（`--stage`／`--interactive`）才會發生。誤判在這裡至多多印一行提示、
> **零資料危害**。詳見 [指令手冊 `memory-merge --fuzzy`](03-指令手冊.md) 與 [已知限制](05-windows-與已知限制.md)。

### 8b. memory 衝突：留兩版到「安全區」＋明文外洩警告
同一則 memory 兩邊改成不同內容時，工具**只讀、絕不寫回你的正式 memory**，
而是把兩個版本另存到**你 memory 資料夾以外**的本機暫存區，再產生一段「請 Claude 幫你合併」的提示詞。

⚠ **重要的明文外洩警告**：
如果你把兩版 memory 貼進 Claude 對話請它合併，**那段對話會被存進 transcript，再被同步到 hub**——
等於你的 memory 內容（可能敏感）從 memory **擴散進了對話記錄、還傳到共用碟**。
所以 `memory-merge` 指令會**警告你**，並支援「**只把提示詞印到畫面／本機暫存**」，讓你**先手動刪掉敏感段**再用。
（詳見 [`memory-merge-from.md`](memory-merge-from.md) 與 [指令手冊](03-指令手冊.md)。）

## 九、doctor——出問題時的自救與體檢

工具附一個 `doctor` 指令，平常可以**純唯讀體檢**（看掛載點在不在、權限、狀態檔健不健康、有沒有檔名撞名），
出狀況時可以：

- `--rebuild-state`：底稿（state）壞了、掉了，從磁碟現況**重建**（重建時**不讀**舊底稿，所以底稿壞掉也救得回來）。
- `--break-lock`：清掉當掉留下的 stale 鎖（只清「確定死掉」的，見上面「四、lock」）。
- `--ack-all` / `--show-acked`：對於「工具永遠無法自動解決、每次都重報」的項目（例如壞掉的檔、撞名的兩份），
  你**審閱過**之後可以標記「知道了、別再吵」——但這**只是把畫面上的提醒藏起來**，
  **絕不會**因此讓工具去自動處理它（結構上不可能）；一旦內容變了會自動重新提醒。

## 小結：這些防線怎麼一起工作

一次 `sync --apply` 背後，大致是這樣層層把關：

1. **dry-run** 先讓你看清楚要做什麼（除非你直接 `--apply`）。
2. 動 hub 前先搶 **lock**，避免多台打架。
3. 每個檔先過**安全閘分類**：壞檔／撞 ID／分岔 → 一律**擋下或問人**，不自動寫。
4. 真要寫時走 **read-verify-write**：寫前重讀比對、寫到暫存再改名、寫後讀回驗證。
5. 往你本機寫一律 **keep-both**（C3：不覆蓋既有）。
6. 刪除走 **tombstone**（A3：不復活、不替你刪）。
7. 沒有 **bootstrap 基線**的專案，寧可擋下要你先建基線。

**每一層的預設都是「不確定就停、就留、就報告」。** 這就是「永不靜默丟資料」的具體長相。
