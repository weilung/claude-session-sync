# 概念與運作：一次同步到底發生什麼事？

> 這篇把「Local／Hub／Remote 是什麼」「你跑一次指令，背後發生什麼」講清楚。
> 讀完你就有正確的心智模型，後面的 [指令手冊](03-指令手冊.md) 和 [情境劇本](04-情境劇本.md) 會很好懂。

---

## 東西存在哪裡？

Claude Code 把每個專案的對話與 memory 存在你電腦的這個位置：

```
~/.claude/projects/<專案資料夾>/
    ├── <sessionId>.jsonl        ← 一則對話 = 一個 .jsonl 檔
    ├── <sessionId>.jsonl
    └── memory/
        ├── MEMORY.md            ← memory 的「目錄索引」
        ├── 某個事實.md          ← 一個事實 = 一個 .md 檔
        └── 另一個事實.md
```

（Windows 上是 `%USERPROFILE%\.claude\projects\...`；若你有設 `CLAUDE_CONFIG_DIR`，就以它為根。）

- **一則對話 = 一個 `.jsonl` 檔**，檔名是那則對話的 `sessionId`。
- **memory = `memory/` 資料夾裡一堆 `.md`**，一檔記一件事；`MEMORY.md` 是它們的索引。
- **專案的定位靠「資料夾名 + sessionId」**，資料夾名是把你的工作路徑（cwd）裡的 `: \ /` 換成 `-` 得到的。

> ⚠ 這個「路徑 → 資料夾名」的轉換**有損**：`a:b`、`a/b`、`a\b` 都會變成 `a-b`，無法反推。
> 所以工具不完全信任資料夾名來判斷「兩邊是不是同一個專案」，另外靠 hub 裡的小記錄檔（sidecar）記下原始路徑。這點在 [名詞對照](06-名詞對照.md) 有解釋。

## 三個角色再說一次（這次講細一點）

### Local（你這台）
就是上面 `~/.claude/projects/`。你**這一台**電腦的對話與 memory。

### Hub（自己這群共用的碟）
一個放在**外接硬碟或網路碟**上的資料夾（例如 `D:\SyncDrive\HomeJSONL`）。
你家幾台電腦**都對它雙向同步**，久而久之大家的內容會**收斂成一致**。

- hub 裡的結構跟 local 很像（`projects/<專案>/*.jsonl` + `memory/`），
  另外多了一些**工具自己的小檔**：sidecar（記專案身分）、`.tombstones/`（刪除標記與鎖）等。
- 「**自己這群共用的那個 hub**」在設定檔裡叫 `own_hub`。日常 `sync` 就是對它。

### Remote hub（別群的碟）
另一群（例如公司）的 hub。你**不會**自動跟它同步。
只有在你**明確下 `pull` / `push` 指令**、而且**可以挑選要哪幾則**時，才會跟它交換。

## 「插上碟、跑一次」——一次 `sync --apply` 的完整流程

假設你在家，外接碟已掛好，跑 `claude-session-sync sync --apply`。背後大致這樣走：

1. **掃描**
   列出你 local 和 hub 上各專案的 `.jsonl` 和 memory，用 `sessionId`／檔名把兩邊配對起來。

2. **安全閘分類**（先擋壞的，再分類）
   逐則判斷這對檔案屬於哪種狀況：

   | 狀況 | 意思 | 工具怎麼做 |
   |------|------|-----------|
   | `identical` | 兩邊一模一樣 | 跳過 |
   | `copy`（單邊有） | 一邊有、一邊沒有，而且是安全的新東西 | 自動複製過去 |
   | `fast-forward`（ff） | 一邊是另一邊的**乾淨延伸**（只多接了新內容在尾巴） | 自動把舊的那端更新上來 |
   | `fork` / `superset-branch` | 兩邊從共同起點**各自長出不同內容**（分岔） | **停下來問你**，不自動選邊 |
   | `damaged` | 檔案壞了、或同一行前後不一致 | **擋下**，列待查，不碰 |
   | `identity-collision` | 根本像是兩則不同對話撞在一起 | **擋下**，請你人工看 |

   > 只有 `identical` / `copy` / `fast-forward` 會**自動**處理；其餘一律**保守：問你或擋下**。

3. **計畫（預設只預覽）**
   把「我打算對每一則做什麼」印出來。**沒加 `--apply` 就到此為止**，什麼都不寫。

4. **落地（有 `--apply` 才做）— 層層把關**
   - 動 hub 前先搶 **lock**（避免多台同時寫）。
   - 每個要寫的檔走 **read-verify-write**：寫前重讀比對、寫到暫存檔再改名、寫後讀回驗證。
   - 往你本機寫一律 **keep-both**（不覆蓋你既有的檔）。
   - 偵測到刪除 → 寫 **tombstone**（不復活、不替你刪別台）。

5. **分岔要你決定的部分**
   如果你加了 `--interactive`，會針對每個 fork 問你「合併 / 留兩份 / 跳過」；
   沒加就只**報告**，留著等你有空處理。

（每一道防線的「怕什麼、怎麼防」細節，見 [安全機制白話](02-安全機制白話.md)。）

## memory 的同步跟對話有什麼不同？

memory 也是雙向同步，但因為它常是你**精煉過、甚至敏感**的資訊，規則更保守：

- **兩邊都有、內容不同** → 不自動合併，報成**衝突**，交給你用 `memory-merge` 處理（見 [指令手冊](03-指令手冊.md)）。
- **一邊刪了** → 走 tombstone，不復活。
- **同一件事、同 `name`、不同檔名** → 當**衝突**報（`conflict-cross-file-identity`），交你用 `memory-merge` 處理。
- **同一件事、不同 `name`（不同 slug）** → 有唯讀輔助 `memory-merge --fuzzy` **列出**疑似重複候選供你複核（**只建議、絕不自動合併**；見 [指令手冊](03-指令手冊.md)）。
- `MEMORY.md`（索引）由工具**機械重建**，但**偵測到你手寫的段落就保留、不動**。

memory 的衝突**不會混進日常 `sync`** 的自動流程——它走**獨立的 `memory-merge` 指令**，
因為衝突合併可能牽涉把內容餵給 Claude（有明文外洩風險），必須跟例行同步分開、由你主動處理。

## 什麼時候「什麼都不會發生」？

這些情況工具會**優雅地不動作**，不是壞掉：

- **碟沒掛**：hub 路徑不存在 → 它就報告掛載點不在，什麼都不寫。
- **第一次、還沒 bootstrap**：沒有基線的專案會被擋下，要你先建基線（見 [安全機制白話 §五](02-安全機制白話.md)）。
- **偵測到分岔但沒加 `--interactive`**：只報告、保留兩份、不動。
- **沒加 `--apply`**：全部只預覽。
