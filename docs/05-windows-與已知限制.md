# Windows 與已知限制

> 這篇講兩件事：① 在 Windows 上用有什麼要注意的；② 目前**還沒做／有邊界**的地方——照實列出，讓你心裡有底。
> 這些限制**都不會弄丟你的資料**（那是最高鐵則），多半是「還沒做的便利功能」或「保守擋下、要你手動處理」。

---

## 一、Windows 上的注意事項

這工具**跨平台都能跑**（Windows / macOS / Linux），且在 Windows 上專門做過一輪「可攜性」強化。幾個重點：

### 設定檔與路徑位置不一樣
- 設定檔：`%APPDATA%\claude-session-sync\config.toml`（不是 `~/.config`）。
- 對話與 memory 根：`%USERPROFILE%\.claude\projects\`（若設了 `CLAUDE_CONFIG_DIR` 就以它為根）。
- 設定檔裡的 Windows 路徑用**單引號**包住，反斜線不用轉義：
  ```toml
  own_hub = 'D:\SyncDrive\HomeJSONL'
  ```

### 換行符已被正確處理（一個曾經的大坑，已修）
早期版本在 Windows 上有個嚴重問題：寫檔時 Windows 會偷偷把換行 `\n` 變成 `\r\n`，
導致「寫出去的內容 ≠ 讀回來的內容」，破壞每個對話檔。
**這個已經修好了**（改用二進位方式寫檔），現在 Windows 上寫出與讀回逐位元組一致。

### 多帳號共用（junction）是被支援的
如果你在同一台機器用多個 Claude 帳號、並用 junction 把 `projects/` 指到同一份真實資料共用——
工具**信任並跟隨** `CLAUDE_CONFIG_DIR` / junction 指定的位置，不會把它誤判成兩份而重複同步。
（只有工具**自己的**暫存／快取區出現可疑的 reparse 才會保守拒絕，那是外洩防線。）

### 有些安全測試在 Windows 上會「跳過」——那是正常的
工具的測試裡有一批是**只有 POSIX（Linux/macOS）才有的行為**（例如某些檔案權限、大小寫不敏感、特殊檔名）。
這些在 Windows 上會顯示 **skipped（跳過）**，不是失敗、不是壞掉——只是那個情境在 Windows 不存在。
這批測試會在 Linux 的自動化 CI 上實跑，確保覆蓋。

## 二、目前已知的限制（照實列）

### 1. 「同一件事、不同檔名」的 memory 模糊比對——**已做，但刻意只到「建議」為止**
- **現況**：自動同步用的是**精確比對**（同 `name`／frontmatter 字面相同才算同一則）。此外多了一個**唯讀輔助工具** `memory-merge --fuzzy`：對「不同 slug、疑似同一事實」的候選對做純字面近似（name＋description 詞元），**列出來讓你人工複核**。
- **刻意的邊界（不是待辦，是設計）**：fuzzy **永遠只建議、絕不裁定**——分數不會自動合併/覆蓋任何 memory；要把兩版拉出來合併，得你**逐對放行**（`--stage`／`--interactive`）。這是因為它動到「**絕不能把兩則意思不同的 memory 判成同一則**」這條最高鐵則，所以設計上把「判定」永遠留給你。
- **會弄丟資料嗎？不會**：誤判在這裡至多**多印一行提示**；沒有你放行，什麼都不會被保留或改動。
- **細節**：見 [指令手冊 `memory-merge --fuzzy`](03-指令手冊.md)。

### 2. 超長路徑（Windows MAX_PATH，260 字元）——**部分支援**
- Windows 傳統上路徑不能超過 260 字元。
- **memory-merge 的暫存路徑**（會很長，因為巢狀又編碼過）**已支援超長**（內部用 `\\?\` 長路徑寫入）。
- **其他非暫存的檔案**（很深的專案路徑下的對話／狀態檔）若超過 260 字元，目前會**保守報錯擋下**（`status='error'`），**不會**寫出一個「寫得進、下次卻讀不回」的壞檔。
- **會弄丟資料嗎？不會**：超長時是 fail-closed 擋下、可見地報錯，不是靜默出錯。完整支援所有超長路徑列為需要時再做。

### 3. 網路碟／外接碟的鎖不是 100% 可靠——**同群同時只跑一台**
- NFS、SMB、exFAT、雲端掛載的檔案鎖與改名原子性**無法保證**。
- **建議用法**：同一群裡，**同一時間只讓一台電腦跑同步**。這條在文件裡明講、不藏。
- 當掉留下的 stale 鎖可以用 `doctor --break-lock` 安全清除（Windows 上也能正確偵測「程序已死」了，這項已完成）。

### 4. `doctor --ack-all` 標記過的「已知問題」若帳本本身壞了
- 你標記「別再吵」的紀錄若檔案損壞，`--unack-all` 清不掉它（要手動刪，或下次標記時自動取代）。
- **不影響安全**：它從不遮蓋**新**問題，內容一變照樣重新提醒。

### 5. 分岔用 `union` 合併之後，那筆分岔**仍會一直被報告**——沒有「已解決」的收斂出口
- **現況**：`union` 把合併結果寫成**新檔**，**絕不覆蓋**你原本那兩個分岔的檔案（這是「絕不覆寫對話原檔」鐵則的直接後果）。因此原檔仍然分岔，`status`／`sync` 之後**每次都還是會報它一次**。
- **這不是失敗**：合併結果好好地在新檔裡、兩邊都有。只是清單上會一直多一行。
- **⚠ 不要重跑 union 想讓它消失**：只會**再生一個重複的合併檔**（時間戳不同）。
- **要它安靜的唯一方法**：自己刪掉 local 的原始檔（保留合併後的新檔）。下次同步會視為「你刪了這則對話」→ 在 hub 寫刪除標記；**hub 上的檔案不會被刪**（保留歸檔、只是不再往回複製）。
- **會弄丟資料嗎？不會**：不處理它也完全安全，純粹是提示噪音。
  （對比：`damaged` / 撞名這類問題有 `doctor --ack-all` 可以標記「別再吵」；**分岔目前沒有**對應機制。）
- **細節**：見 [情境劇本 §情境 4](04-情境劇本.md)。

### 6. 一些「理論上存在、實際上碰不到」的極小邊界
文件裡誠實記錄了一些**有界殘留**——都是「需要對抗式的檔案系統操作、或微秒級的時間差、或超長路徑＋特殊 symlink 同時發生」才可能觸發，在正常單人使用下碰不到，而且**寫入前都已 fail-closed**（不會走到覆蓋／復活／丟失）。這些屬於嚴謹規格的範圍，記在 `DESIGN.md` 與 `HANDOFF.md`，日常使用不需擔心。

## 三、一句話總結

- **Windows 完全能用**，且踩過的跨平台坑（換行符、超長暫存路徑、死鎖偵測）都已修。
- memory 的「同事實不同檔名」模糊比對**已做**，但刻意**只到「列出候選、由你放行才保留兩版」為止**、絕不自動合併——**不會弄丟資料**。
- 所有已知限制的共同點：**寧可多留、多問、多擋，也不靜默覆蓋。**
